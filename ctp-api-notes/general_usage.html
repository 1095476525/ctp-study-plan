General API Usage

User can use CThostFtdcTraderApi and CThostFtdcMdApi to send the request to server, and get the response via implement the interface CThostFtdcTraderSpi and CThostFtdcMdSpi.
Generally, the request of:

int CThostFtdcTraderApi::ReqXXX(CThostFtdcXXXField *pReqXXX, int nRequestID)
// or:
int CThostFtdcMdApi::ReqXXX(CThostFtdcXXXField *pReqXXX, int nRequestID)

should get the response of:

void CThostFtdcTraderSpi::OnRspXXX(CThostFtdcXXXField *pRspXXX, CThostFtdcRspInfoField *pRspInfo, int nRequestID, bool bIsLast);
// or:
void CThostFtdcMdSpi::OnRspXXX(CThostFtdcXXXField *pRspXXX, CThostFtdcRspInfoField *pRspInfo, int nRequestID, bool bIsLast);

, where 'XXX' can be replaced by the action name, e.g. 'UserLogin'.

Client requests

int CThostFtdcTraderApi::ReqXXX(CThostFtdcXXXField *pReqXXX, int nRequestID)
// or:
int CThostFtdcMdApi::ReqXXX(CThostFtdcXXXField *pReqXXX, int nRequestID)
  • The first parameter of the request function is a pointer to an instance of a data structure, which should not be NULL and the fields of the structure needs to be set with correct values(for some fields, empty value is also acceptable).
  • The second parameter of the request is the request id, which should be maintained by the api user. The request id is used for the api user to recognize different requests/responses, it is strongly recommended to keep the request id unique within the session.
    Return values
    0Success.
    -1Network connection failure.
    -2Exceeded the limit of max quantity of buffered(unhandled) requests.
    -3Exceeded the limit of max requests per second.
    Note
    A 0(success) return value only means the local function call is succeed and the request will be sent to the server through the network, it doesn't mean the the request is accepted by server, user can get the server response through the callback functions: OnRspXXX(), OnRtnXXX() and OnErrRtnXXX().

Server responses

void CThostFtdcTraderSpi::OnRspXXX(CThostFtdcXXXField *pRspXXX, CThostFtdcRspInfoField *pRspInfo, int nRequestID, bool bIsLast);
// or:
void CThostFtdcMdSpi::OnRspXXX(CThostFtdcXXXField *pRspXXX, CThostFtdcRspInfoField *pRspInfo, int nRequestID, bool bIsLast);
  • The first parameter is the data of response, which usually includes the original request data. If something is wrong or the server cannot find any record for the request, this parameter will be set to NULL.
  • The second parameter 'pRspInfo' contains two fields, one for error code and one for error message. If the request is succeed, error code will be set to 0, otherwise a non-0 error code is returned. Please note that one callback function can be invoked more than one time for a single request, only the first callback may have the none-null 'pRspInfo' parameter.
  • The third parameter 'nRequestID' is the request id, which is same to the corresponding request.
  • The last parameter 'bIsLast' indicates if the current response is the last reponse of the same request. Sometimes the callback function needs to be called multiple times (with same 'nRequestID') to return the full list of data, only the last callback will have the 'bIsLast' parameter to be set to true.

Server notifications

When the server needs to publish notifications(e.g. new order status) to the api client, the following functions will be called:

// For trader api only:
void CThostFtdcTraderSpi::OnRtnXXX(CThostFtdcXXXField *pXXX);
void CThostFtdcTraderSpi::OnErrRtnXXX(CThostFtdcXXXField *pXXX, CThostFtdcRspInfoField *pRspInfo);
// For quotation api only:
void OnRtnDepthMarketData(CThostFtdcDepthMarketDataField *pDepthMarketData);

, where 'XXX' can be replaced by the event name, e.g. 'Trade'.

BrokerID, InvestorID and UserID

There are three IDs in the api: BrokerID, InvestorID and UserID. The server will check these IDs with client's requests, the IDs should keep consistent during the session. e.g. If a client logon succeed with BrokerID="broker1", InvestorID="user1", UserID="user1", and then sends an order insert request with BrokerID="broker1", InvestorID="user2", UserID="user2", the server will reject the request because the InvestorID and UserID are changed.

Note
The following are the mapping of the three IDs to ATP account name and user name:
  • BrokerID is mapped to account name of ATP.
  • InvestorID is mapped to user name of ATP.
  • UserID is mapped to user name of ATP.
Sample code
static const char *BROKER_ID = "test_broker_id";
static const char *USER_ID = "test_user_id";
static const char *INVESTOR_ID = USER_ID; // investor id is mapped to user id in ATP.
void requestOrderInsert() {
CThostFtdcInputOrderField field;
memset(&field, 0, sizeof(field));
strcpy(field.BrokerID, BROKER_ID);
strcpy(field.InvestorID, INVESTOR_ID);
strcpy(field.UserID, USER_ID);
strcpy(field.InstrumentID, "C1803-DC");
strcpy(field.OrderRef, "1");
// ...
int rtnCode = tradeApi->ReqOrderInsert(&field, requestId);
if (rtnCode != 0) { Logger::info("[ERROR] [%s:%s:%3d] Request failed: code=%d", __FILE__, __FUNCTION__, __LINE__, rtnCode); }
}
