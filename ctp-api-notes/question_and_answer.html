Q&A

How can I get currency, rate and lot size?

API reuses the field ExchangeInstID of structure CThostFtdcInstrumentField in callback CThostFtdcTraderApi::ReqQryInstrument() to provide the infomation.
The field CThostFtdcInstrumentField::ExchangeInstID is a null-terminated string which contains the currency code(ISO 4217 standard), rate(against USD) and lot size of the instrument, separated by comma, e.g. "CNY,6.638,5".

How to handle spread order?

  1. The spread symbol is formatted as SP_<CONTRACT-CODE>_<MONTH1>_<MONTH2>-<EXCHANGE>, the leg with MONTH1 always has the same direction as the spread symbol.
    e.g. Buy SP_UC_1809_1806-SGX can be explained as Buy UC1809-SGX and Sell UC1806-SGX. Sell SP_UC_1809_1806-SGX can be explained as Sell UC1809-SGX and Buy UC1806-SGX.
  2. User can create a spread order with CThostFtdcTraderApi::ReqOrderInsert() as normal orders.
  3. When the spread order is traded, the api CThostFtdcTraderSpi::OnRtnTrade() will be called twice with the trade info of each leg.
  4. The side of the leg can be identified by the direction of the individual trade. User can also identify the direction of the leg by comparing the leg symbol name with the spread symbol name, as explained before.
    e.g. If triggered a spread order: Buy SP_UC_1809_1806-SGX, two trade will come subsequently(if traded): UC1809-SGX with direction 'Buy' and UC1806-SGX with direction 'Sell'.
  5. User should not use the trade id to identify the side of leg even if the trade id is encoded as 'XXXXA' or 'XXXXB'(the suffix 'A' and 'B' means the side of the leg).
    The trade ids are come from different trading systems, each system has its own algorithm to generate the trade id, so user should not assume that all the trade ids are
    encoded with the same format.

Why I got -2 return code with the API call?

The api dll only allows one concurrent query request(exclusive of ReqOrderInsert and ReqOrderAction), the new query api call will return -2 if the previous one is not yet responsed.

Why I got -3 return code with the API call?

Only limited number of queries(exclusive of ReqOrderInsert and ReqOrderAction) can be accepted per second, -3 will be returned if there're too many queries within a second. By default, only 1 query is allowed per second.

How to handle SIGPIPE signal?

The api dll needs to create TCP connections to communicate with server, and SIGPIPE signal may be introduced while writing data to a socket. By default, the operating system will terminate the client program if SIGPIPE signal
is generated, client need to add code to change the default behaviour of handling SIGPIPE. The following code needs to be added before creating any api instances:

struct sigaction sa; // defined in signal.h
sa.sa_handler = SIG_IGN;
sigaction(SIGPIPE, &sa, 0);

The dll does not catch the SIGPIPE signal internally is because the signal and it's handler are shared by the whole process, not only the dll library. Client may need to add customized handler to process the signal. If the dll
registered a handler of the signal, it may overwrite the client customized handler, which may causing unexpected issues.
